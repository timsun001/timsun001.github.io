3 真正的难点：Feedback Loop (20min)

技术壁垒越来越低，真正的门槛在企业内部怎么让知识库转起来。

3.1 知识库的死因

上线 → 初期新鲜感 → 信息过期无人更新 → 用户问不到答案 → 不再使用 → 废弃
  • 信息沉淀难 — 核心知识在人的脑子里（"这个问老王"），不在文档里
  • 更新滞后 — 文档半年前过期，没人标记、没人更新、没人删
  • 无反馈回路 — 用户问了个问题，系统给了错误/过期的回答，没人纠正
  • 参与激励缺失 — 员工为什么要贡献知识？KPI 里没这项

3.2 如何建立 Feedback Loop？

用户提问 → AI 回答 → 满意度打分/纠错
                ↓
    [低分/纠错] → 通知知识负责人 → 更新文档 → 知识库优化
                ↓
    自动触发：重复纠错 → 模型微调 / RAG 策略调整

① 谁来维护知识？ 专职 KA vs 众包模式 · 权限审核 · 自动 freshness check

② 怎么让员工用？ 飞书/钉钉/企微机器人入口 · 反馈收集机制 · 使用数据可视化

③ 怎么让员工贡献？ 被问到问题时一键录成 FAQ · 积分/KPI 激励 · 聊天记录一键入库

3.3 分级作战策略

层级做法人工介入程度覆盖场景
L1 文档差上传 PDF/Word → RAG 问答高：大量人工清洗文档、标注规章制度、产品手册、培训资料
L2 有人维护Wiki/飞书文档 → 实时同步中：依赖人主动更新维护SOP、FAQ、项目文档
L3 主动运营专人维护 + dashboard + 定期 audit中低：专职 KA 运营，使用数据驱动客户服务、技术支持、合规
L4 自动进化用户反馈闭环 + 自动知识提取 + AI 梦境整合极低：AI 自主运作，人为抽查兜底头部客户，需持续优化
企业的知识管理水平决定了 AI 知识库的上限，而不是技术选型。一个连飞书文档都维护不好的企业，上 RAG 也是白搭。

3.4 技术参考（一句话）

文档 → 解析/OCR → 分块 → 向量化 → 存储
用户提问 → [向量搜索 + 关键词搜索] → 重排序 → LLM → 答案
组件推荐
EmbeddingBGE-M3 (BAAI, 多语言最强)
向量库Qdrant (混合搜索最快)
LLMDeepSeek-V4 / Qwen2.5
编排LangChain + LangGraph
文档解析PaddleOCR + Unstructured.io

重点：技术选型不是差异化，产品和运营才是。 Yunzhao 的公司作为数据产品公司，核心价值应该是对客户的行业理解、实施能力、持续运营——这些才是大厂懒得做也做不好的。

🎯 核心结论（备说）

  1. 市场确实在，窗口期 ~12-18 个月
  2. 技术门槛不高，真正的难点是企业的知识管理水平和反馈闭环
  3. 用开源栈（BGE-M3 + Qdrant + DeepSeek-V4）快速迭代，零授权费

  5. 潜在风险：大厂（阿里/腾讯/字节）可能做，但它们的问题是产品太重、定制化差
